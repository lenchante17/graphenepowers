## 한 줄 멘탈모델

GraphenePowers는 "개발 잘하는 팁 모음"이 아니라, 요청을 받아서 완료까지 보내는 **작업 운영체계**에 가깝다.

조금 더 정확히 말하면:

- 입력은 사용자 요청이다.
- 첫 동작은 구현이 아니라 분류다.
- 분류 결과에 따라 설계, 계획, 실행, 리뷰, 회고의 경로가 달라진다.
- 실행 중에도 상태 전이가 명시되어 있고, 실패 시 되돌아갈 경로가 정해져 있다.
- 최종 목표는 "코드를 썼다"가 아니라 "증거를 가진 상태로 끝냈고, 다음 작업이 더 좋아지도록 학습했다"이다.

이 점에서 이 시스템은 애자일 체크리스트보다는 **상태 기계 기반 delivery workflow**에 더 가깝다.

## 핵심 설계 철학

문서 전반에서 반복되는 메시지는 아래 다섯 가지다.

1. 작업 복잡도에 비례하는 절차를 써야 한다.
2. 구현보다 먼저 분류와 설계를 해야 한다.
3. 계획은 순서 목록이 아니라 의존성 그래프여야 한다.
4. 검증과 리뷰는 구현자의 확신이 아니라 독립적인 단계여야 한다.
5. 작업이 끝나면 결과보다 프로세스 편차를 먼저 돌아봐야 한다.

즉, GraphenePowers는 "빠르게 뭔가 만드는 법"보다 "작업을 제어 불가능하게 망치지 않는 법"에 더 많은 비중을 둔다.

## 워크플로우 단계별 해석

### 1. Entry: `using-graphenepowers`

이 엔트리 스킬은 사실상 워크플로우의 헌법 역할을 한다.

핵심 메시지는 단순하다.

- 작업 전에 스킬을 확인하라.
- "이건 간단하니까 예외"라는 합리화를 금지하라.
- 개발 작업은 기본적으로 `triage`를 거쳐라.

즉, GraphenePowers는 자유로운 즉흥 실행보다 **초기 라우팅의 일관성**을 더 중요하게 본다.

### 2. Classification: `triage`

`triage`는 이 시스템의 가장 실용적인 출발점이다.

이 스킬은 다섯 차원으로 작업을 점수화한다.

- 범위
- 불확실성
- 결합도
- 외부 의존성
- 리스크/비용

여기서 중요한 것은 합산보다 `hard override`다.

이 설계는 매우 의도적이다.

- 전체 평균이 낮아도 특정 위험 하나가 크면 더 무거운 절차로 올린다.
- `confidence`와 `profile asymmetry`를 분리해, "확신이 낮은 것"과 "특정 리스크가 튀는 것"을 다른 문제로 본다.

이건 일반적인 태스크 분류보다 한 단계 더 정교하다. 문서 기준으로 보면 GraphenePowers는 "작업이 얼마나 큰가"보다 "작업을 너무 쉽게 봐도 되는가"를 묻는다.

### 3. Design path: `brainstorming`

Feature급 작업은 바로 계획으로 가지 않고 설계 대화로 들어간다.

이 단계의 존재는 GraphenePowers가 구현 실패의 상당 부분을 "코딩 실수"보다 **설계 불명확성**에서 온다고 본다는 뜻이다.

좋은 점은 다음과 같다.

- 설계와 구현을 분리한다.
- 사용자 승인 없는 구현을 금지한다.
- 아이디어 수준의 요청을 곧바로 작업 계획으로 착각하지 않게 만든다.

대신 이 단계는 문맥 전환 비용이 있다. 작은 실험성 작업까지 자주 Feature로 올라가면 체감상 매우 무거워질 수 있다.

### 4. Planning path: `writing-plans`

이 스킬은 GraphenePowers의 본체 중 하나다.

문서상 계획은 단순 TODO가 아니라:

- DAG
- CPM
- dual-PERT
- task card
- kanban-friendly state

를 포함하는 실행 그래프다.

이 구조는 두 가지를 동시에 노린다.

- 사람이 읽기 쉬운 Markdown 계획
- 실행 중 상태 전이를 기록할 수 있는 운영 데이터

즉, 계획 문서는 사전 설명서가 아니라 나중에 실행과 리뷰가 참고할 **정본 데이터 구조**가 된다.

이 설계는 매우 강력하지만, 반대로 계획 품질이 낮으면 모든 후속 단계가 약해진다.

### 5. Execution path: `executing-plans`

이 단계가 GraphenePowers를 평범한 "좋은 개발 습관 모음"과 가장 크게 구분한다.

실행은 자유형이 아니라 다음 상태 기계로 정의된다.

- `prepare`
- `dispatch`
- `track`
- `unblock`
- `verify`
- `close`

여기서 중요한 멘탈모델은 "개발자 = 수행자"가 아니라 "오케스트레이터 = 상태 관리자"라는 점이다.

문서 기준으로 실행 단계는 다음 원칙을 갖는다.

- `plan-progress.md`는 단일 writer만 갱신한다.
- 병렬 실행은 `write_set` 충돌이 없을 때만 허용한다.
- blocker가 생겨도 전체 작업을 멈추지 않는다.
- 검증은 성공 주장 전에 반드시 거쳐야 하는 상태다.

이 구조는 인간 팀 운영과 에이전트 오케스트레이션을 동시에 염두에 둔 설계처럼 보인다.

### 6. Debugging and TDD as subroutines

`systematic-debugging`과 `test-driven-development`는 별도 워크플로우라기보다, 메인 파이프라인 안으로 호출되는 보조 규율이다.

의도는 명확하다.

- 실패 원인이 불명확하면 무작정 수정하지 말고 디버깅 절차로 내려가라.
- 코드 변경이 필요하면 구현은 TDD 규율 안에서 하라.

즉, GraphenePowers는 "예외 상황"도 임시방편으로 처리하지 않고, 이미 정의된 하위 절차로 재귀적으로 흡수하려 한다.

### 7. Review and closure: `code-review` + `retrospective`

`code-review`는 하나의 이름 아래 여러 리뷰 모드를 둔다.

- Preflight Spec Review
- Spec Delta Review
- Quality Review

이 구조는 "리뷰 = 마지막 코드 확인"이 아니라:

- 실행 전에 계획 자체를 검토하고
- 실행 중 드리프트를 검토하고
- 마지막에 품질을 검토하는

다층 방어선을 만들겠다는 뜻이다.

`retrospective`는 더 중요하다.

GraphenePowers는 여기서 작업 종료를 "머지 여부"보다 "이번 실행이 다음 실행을 어떻게 바꿀지"로 본다.

즉, 이 시스템은 산출물 중심이면서 동시에 **프로세스 학습 시스템**이기도 하다.

## v4에서 보이는 구조적 변화

마이그레이션 문서를 보면 v4는 기능을 계속 늘린 것이 아니라, 의미가 겹치는 스킬을 정리해 **공개 인터페이스를 줄인 버전**에 가깝다.

핵심 변화는 다음처럼 읽힌다.

- `triage`를 정식 엔트리 분류기로 승격
- `writing-plans`와 `executing-plans`를 중심축으로 재구성
- 리뷰를 `code-review` 하나로 통합
- 레거시 스킬은 deprecated wrapper로 남겨 하위 호환성 유지

즉, v4의 방향은 "스킬을 많이 두는 것"이 아니라 "실제 운영 상태를 가지는 몇 개의 중심 스킬로 모으는 것"이다.

이건 좋은 방향이다. 워크플로우 시스템은 표면 API가 커질수록 실제로는 더 약해지기 쉽기 때문이다.

## 이 워크플로우의 예상 장점

### 절차를 난이도에 맞게 조절한다

모든 일을 같은 무게로 다루지 않는다. 작은 일은 빠르게, 큰 일은 설계와 계획을 붙여서 다루려는 점이 실무적이다.

### 계획과 실행 사이의 단절을 줄인다

많은 팀에서 계획 문서는 버려지고 실행은 별도로 흘러간다. GraphenePowers는 계획 자체를 실행 데이터 구조로 쓰려 하기 때문에 이 단절을 줄일 가능성이 있다.

### 병렬 실행에 강한 멘탈모델이 있다

`depends_on`, `write_set`, blocker, task card, orchestrator 같은 개념은 여러 작업이 동시에 움직일 때 특히 강해진다.

### "증거 없는 성공 주장"을 막는다

`verify` 상태, 리뷰 분리, retrospective를 보면 구현자의 자기 보고보다 검증 가능한 결과를 더 우선한다.

### 실패를 프로세스 개선 기회로 본다

보통 회고는 팀 문화 문서에서만 존재하는데, 여기서는 워크플로우 자체 안에 들어가 있다. 이건 좋은 설계다.

## 이 워크플로우의 예상 약점

### 문서 오버헤드가 크다

작업이 작거나 맥락이 단순한데도 문서를 과하게 요구하면, 사용자는 워크플로우를 우회하려 들 가능성이 높다.

### 오케스트레이터 병목 위험이 있다

`plan-progress.md` 단일 writer 모델은 정합성에는 좋지만, 실무에서는 기록 책임자가 전체 흐름의 병목이 될 수 있다.

### 계획 품질에 지나치게 의존할 수 있다

`write_set`, acceptance, verification, task decomposition이 부실하면 시스템은 형식만 남고 실질은 사라진다.

### discovery보다는 delivery에 치우쳐 있다

이 시스템은 "무엇을 왜 만들까"보다 "정해진 일을 어떻게 통제 가능하게 만들까"에 더 강하다. 제품 전략이나 우선순위 결정까지 포괄하는 프레임은 아니다.

### 규율이 내면화되지 않으면 관료주의가 된다

문서만 읽고 예상해보면 가장 큰 실패 모드는 이것이다.

- 사람은 체크리스트를 채운다.
- 하지만 실제 판단은 하지 않는다.
- 결과적으로 문서는 늘고 품질은 그대로다.

강한 프로세스는 강한 사고와 같이 갈 때만 효과가 있다.

## 예상되는 실패 모드

문서에서 직접 말하지는 않지만, 구조상 다음 위험이 보인다.

### 1. `triage`가 지나치게 보수적으로 작동하는 경우

많은 작업이 `Task`나 `Feature`로 올라가면 팀은 시스템을 무겁다고 느낄 것이다.

### 2. `plan-progress.md`가 관리 대상이 아니라 의식이 되는 경우

상태 기록이 실제 운영 데이터가 아니라 "나중에 채우는 보고서"가 되면 전체 모델이 무너진다.

### 3. 리뷰 단계가 품질 필터가 아니라 큐 적체 지점이 되는 경우

리뷰가 강할수록 좋다는 보장은 없다. 리뷰 대기 시간이 길어지면 흐름 전체가 느려진다.

### 4. retrospective가 형식적 사후 정리로 전락하는 경우

실패 패턴을 실제로 다음 스킬 개정이나 규칙 개선에 반영하지 않으면 회고는 장식이 된다.

### 5. 사람이 아니라 에이전트에게만 최적화되는 경우

문서 구조가 너무 기계 친화적이면 인간 사용자는 피곤해질 수 있다. 반대로 인간 친화성만 높이면 자동화 가능성이 약해진다. GraphenePowers는 두 세계 사이 균형을 계속 맞춰야 한다.

## 어떤 상황에 잘 맞는가

- 여러 작업이 동시에 움직이는 중간 이상 규모의 개발
- 병렬 에이전트나 여러 기여자가 섞이는 작업
- 변경 추적, 검증 증거, 리뷰 독립성이 중요한 작업
- 설계 불명확성 때문에 실행이 자주 흔들리는 환경
- "빨리 시작"보다 "중간에 덜 흔들리는 것"이 더 중요한 환경

## 어떤 상황에는 과할 수 있는가

- 개인이 짧게 수정하는 단일 파일 변경
- 하루 안에 끝나는 사소한 자동화
- 탐색적 프로토타이핑
- 빠른 실험이 더 중요한 초기 discovery 작업

즉, GraphenePowers는 항상 좋은 워크플로우라기보다 **복잡성과 병렬성이 일정 수준 이상일 때 빛나는 워크플로우**로 읽힌다.

## 비교해볼 만한 좋은 워크플로우

### Shape Up

닮은 점:

- Feature 전에 설계와 문제 framing을 분리한다.
- 구현 전에 경계를 먼저 세운다.

다른 점:

- Shape Up은 product betting과 appetite가 강하다.
- GraphenePowers는 delivery control과 execution state가 더 강하다.

좋은 참고 포인트:

- Feature 전에 무엇을 하지 않을지도 함께 명시하는 습관

### Kanban

닮은 점:

- `ready / in_progress / blocked / review / done` 같은 lane 사고
- blocker와 흐름 중심 사고

다른 점:

- GraphenePowers는 칸반보다 훨씬 강한 선행 분류와 설계 게이트를 둔다.
- 단순 시각화가 아니라 계획 그래프와 상태 기계를 같이 쓴다.

좋은 참고 포인트:

- WIP limit와 cycle time 측정 규율

### CPM / PERT 기반 프로젝트 관리

닮은 점:

- 의존성 중심 계획
- duration과 effort 구분
- 크리티컬 패스 의식

다른 점:

- 전통적 CPM/PERT보다 훨씬 소프트웨어 실무 친화적이다.
- task card, review, blocker, write-set 같은 운영 개념이 더 많다.

좋은 참고 포인트:

- 추정 오차를 실제 retrospective와 연결하는 방식 강화

### RFC/ADR + PR review

닮은 점:

- 설계 문서를 먼저 만들고 구현을 나중에 한다.
- 구현자와 분리된 리뷰를 둔다.

다른 점:

- GraphenePowers는 이를 단발 문서 프로세스가 아니라 반복 가능한 스킬 흐름으로 통합한다.

좋은 참고 포인트:

- 설계 변경 이력과 결정 이유를 더 명확히 남기는 구조

### SRE incident / postmortem discipline

닮은 점:

- 원인 불명 실패는 임시 수정으로 덮지 않는다.
- 회고를 학습 장치로 본다.

다른 점:

- SRE는 운영 사고 대응에 더 가깝고, GraphenePowers는 일반 개발 전 과정을 포괄하려 한다.

좋은 참고 포인트:

- blame 없는 회고와 재발 방지 액션의 추적성

### Temporal 같은 상태 기계형 오케스트레이션

닮은 점:

- 명시적 상태
- 실패 시 복귀 경로
- 장기 실행 플로우를 제어하는 사고

다른 점:

- Temporal은 런타임 워크플로우 엔진이고, GraphenePowers는 개발 프로세스용 운영 모델이다.

좋은 참고 포인트:

- 상태 전이와 이벤트 기록의 일관성 검증

## 종합 판단

문서만 기준으로 보면 GraphenePowers의 본질은 다음과 같다.

- Agile 방법론의 변형이라기보다
- Kanban, CPM/PERT, RFC, 독립 리뷰, 회고를
- 에이전트 시대에 맞게 하나의 운영 모델로 통합하려는 시도

좋은 점은, 이 시스템이 "좋은 습관" 수준을 넘어서 **명시적 통제 모델**을 가지려 한다는 것이다.

위험한 점은, 통제 모델이 실제 실행 데이터와 연결되지 않으면 곧바로 **문서화된 관료주의**가 된다는 것이다.

따라서 이 워크플로우의 성공 조건은 명확하다.

1. `triage`가 과도하게 무겁지 않아야 한다.
2. 계획 문서가 실제 실행 데이터 구조로 살아 있어야 한다.
3. 리뷰와 회고가 큐 적체나 형식주의가 아니라 실제 학습으로 이어져야 한다.
4. 사람과 에이전트 모두에게 이해 가능한 수준의 단순성을 유지해야 한다.

- 이 시스템이 개인 개발자와 다중 에이전트 환경에서 각각 어떻게 다르게 작동해야 하는가
